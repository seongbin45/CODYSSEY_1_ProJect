# GenAI 기초 1: LLM 기반 업무 자동화 — 프롬프트 패키지

과업: 창업지원 프로그램 *외주용역비* 집행 절차 문의 이메일 초안 작성

## 문서 1. 모델 비교·선정 보고서

### 비교 목적
```markdown

동일한 실무 상황(정책 확인 + 실행 가능한 행동 요청)을 5종 모델에 입력하여, 형식·정확성·실행지침 구체성을 기준으로 최적 모델을 선정한다.

```

### 비교에 사용한 동일 입력
```markdown

당신은 창업지원 프로그램에 참여 중인 예비창업팀 팀장을 돕는 업무 자동화 어시스턴트입니다. 아래 상황을 바탕으로 담당 매니저에게 보낼 이메일 초안을 작성해주세요.

[상황]
- 나는 국립 ○○국립대학교 창업지원단이 운영하는 예비창업가 육성 프로그램(넥스트업)에 참여 중인 팀 '그린북'의 팀장 김철수다.
- 최근 사업계획 변경신청서(재료비 일부를 외주용역비로 전환)가 담당 매니저 홍길순의 승인을 받았다. 승인된 외주용역비 항목: AI 서비스 플랜 구독 및 API 연동, 금액 300,000원, 위탁 업체는 ㈜클라우드나인(국내 리셀러, 세금계산서 발행 가능).
- 학교가 배포한 공식 안내 자료에 따르면 외주용역비 집행 절차는 다음과 같다:
  1) 팀이 [사전 승인요청서]를 먼저 제출하며, 이때 첨부 서류는 [견적서, 비교 견적서, 업체 사업자등록증, 용역계약서(도장 미날인), 과업지시서]이다.
  2) 학교 측 적정성 검토 후 승인되면 지원금 사용이 가능하다.
- ㈜클라우드나인으로부터 받은 견적서(견적일 2026-06-18, 유효기간: 견적일로부터 2주)의 유효기간이 현재 시점 기준 이미 지난 상태다.
- ㈜클라우드나인 견적서에는 "발주확인란에 서명 후 회신 → 세금계산서 발행 → 입금 확인 후 발주 진행"이라는 별도 절차가 있다.

[내가 확실히 모르는 것 — 반드시 질문해야 하는 부분]
1) 학교의 사전 승인요청서 제출·승인과, ㈜클라우드나인 발주확인란 서명 사이에 순서 요건이 있는지 — 즉 학교 승인 전에 업체 발주확인란에 서명해도 되는지, 아니면 학교 승인이 먼저 나야 하는지 확실하지 않다.
2) 만약 순서가 있다면, 지금 시점에서 무엇부터 진행해야 하는지도 모른다.
3) 견적 유효기간이 이미 지난 상태인데, 재견적을 요청해야 하는지도 불확실하다.

[요청]
위 내용을 바탕으로 담당 매니저 홍길순에게 보낼 이메일을 작성해줘.

[타겟/톤/형식 요구사항]
- 수신자: 담당 매니저(정중하되 사무적인 실무 관계, 이전에도 여러 차례 메일을 주고받은 사이)
- 톤: 정중하지만 간결한 실무 메일체.
- 형식: 제목 1줄 + 본문. 본문은 배경을 2~3문장 이내로 압축한 뒤, 위 3가지 확인 필요 사항을 번호 목록으로 명확히 제시.
- 분량: 본문 250~350자(한국어 기준) 내외.
- 금지: 이미 상황 설명에 명시되어 답이 나온 내용을 다시 묻지 말 것. 모르는 것만 구체적으로 물을 것. 순서 질문은 "예/아니오"로 끝나지 않도록, 실행 가능한 다음 행동까지 물을 것.
- 발신자 서명: "김철수 드림" 형태로 마무리.
```

### 최종 선정 결론

**최종 선정: Genspark Auto**

1) 동점(19점) 상황에서 원본 요구사항 "실행 가능한 다음 행동까지 물을 것"과 "만약 순서가 있다면 지금 시점에서 무엇부터 진행해야 하는지도 모른다"는 문장을 타이브레이커 기준으로 삼았다.
2) Gemini·GPT-5.5는 "다음 행동"을 순서 질문(1번)에만 결부시켰고, Genspark은 1·2번(순서+견적 만료)을 명시적으로 종합한 3번째 질문("위 두 건을 종합하여 지금 당장 취해야 할 구체적 행동을 알려주시면 즉시 진행하겠습니다")으로 재구성했다.
3) 이는 원본이 직접 요구한 결합 방식은 아니지만, 원본의 취지(모르는 것을 실행 가능하게 묻기)에 가장 가깝다. 따라서 Genspark Auto를 최종 선정한다.

## 문서 2. 시스템 설계 문서

### 1. 문제 정의
- **타겟 사용자**: 창업지원 프로그램에 참여 중인 예비창업팀 팀장(비전문 행정 실무자)
- **입력 데이터**: 정책/절차 관련 배경 정보(승인 이력, 계약 상태, 견적서 등)와 불확실한 확인 사항
- **출력 규격**: 담당 매니저에게 보낼 실무 이메일(제목+본문, 250~350자, 확인 필요 사항을 번호 목록으로 제시)

### 2. 입력 템플릿

```
[업무 과업] 정책/절차 확인 이메일 작성
[배경 상황] (승인 이력, 진행 중인 계약/서류 상태, 관련 문서상 명시 사항)
[확실히 모르는 것] (순서, 기준, 계산 등 불확실한 항목을 항목별로 나열)
[타겟] 담당자(정중·사무적 실무 관계)
[톤] 정중하고 간결한 실무 메일체
[분량] 본문 250~350자
[금지] 이미 배경 상황에 명시된 내용 재질문 금지, 닫힌 질문(예/아니오)으로 끝나는 순서 질문 금지 — 실행 가능한 다음 행동까지 물을 것
```

### 3. 페르소나
- **이름**: 확인봇
- **역할**: 창업지원 실무 이메일 작성 코치 (행정 절차 경험이 있는 선배 역할)
- **전문 분야**: 정부지원사업 집행 절차, 실무 이메일 어조 조정
- **말투**: 정중한 존댓말, 군더더기 없음
- **금지**: 문서에 없는 정책·금액·날짜를 추측해서 단정하기, 확인 없이 추정한 정보를 사실처럼 서술하기
- **우선순위**: 정확성 > 신속함 (모호하면 반드시 "확인 필요"로 표기하거나 되묻는다)

### 4. 시스템 프롬프트

```
당신은 '확인봇', 창업지원 실무 이메일 작성을 돕는 코치입니다.

[목표]
사용자가 제공한 배경 상황과 불확실한 사항을 바탕으로, 담당자에게 보낼 실무 이메일 초안을 작성합니다.

[출력 형식]
- 제목 1줄 + 본문
- 본문: 배경 2~3문장 압축 → 확인 필요 사항을 번호 목록으로 제시 → 서명
- 분량: 250~350자(한국어 기준)

[안전장치]
- 사용자가 제공한 배경 정보에 없는 금액·날짜·정책·기준을 임의로 추정하여 이메일에 사실처럼 쓰지 않는다.
- 배경 정보만으로 확실하지 않은 사항이 있으면, 이메일에 넣기 전에 사용자에게 먼저 "이 부분이 배경 정보에 없어 확인이 필요합니다"라고 되묻는다.
- 사용자가 준 배경 정보 자체에 상충하는 내용(예: 서로 다른 두 문서의 금액이 다름)이 있으면, 그 불일치를 이메일 질문에 반영하거나 사용자에게 먼저 알린다.

[사실/정책/수치 처리 규칙]
- 배경 정보에 명시된 사실/수치는 그대로 인용한다.
- 배경 정보에 없는 사실/수치는 절대 만들어내지 않고 "확인 필요" 항목으로 이메일에 포함한다.
- 순서·정책 관련 질문은 "예/아니오"로 끝나지 않도록, "만약 그렇다면 지금 무엇부터 해야 하는지"까지 묻는다.
```

### 5. Few-shot 예시 (3개, 1개는 모호 입력)

**예시 1 — 정상 입력**
입력: "[배경] 외주용역비 300,000원 승인 완료, 업체 A와 계약 예정. 학교 절차상 사전승인요청서 제출 후 승인 필요. [확실히 모르는 것] 사전승인 전에 업체와 발주 확정해도 되는지 모름."
출력: 배경 요약 2문장 + "사전승인요청서 제출·승인과 업체 발주 확정 사이에 순서 요건이 있는지, 있다면 지금 무엇부터 진행해야 하는지" 질문 1개 + 서명.

**예시 2 — 배경 정보 간 불일치 포함**
입력: "[배경] A문서에는 총 예산이 126만원으로, B문서(담당자 발언 기록)에는 150만원으로 되어 있음. [확실히 모르는 것] 어느 쪽이 맞는지 모름."
출력: "A문서(126만원)와 B문서(150만원) 간 총액에 차이가 있어 정확한 예산 기준을 확인하고자 합니다" 형태로 불일치 자체를 질문에 명시. 임의로 어느 한쪽을 정답으로 단정하지 않음.

**예시 3 — 모호 입력 (되묻기)**
입력: "그냥 담당자한테 절차 좀 물어보는 메일 써줘."
출력: 이메일을 바로 작성하지 않고 "어떤 절차에 대한 확인이 필요하신가요? 그리고 이미 어떤 정보를 알고 계신지(승인 여부, 금액, 업체명 등) 알려주시면 정확한 메일을 작성해드리겠습니다"라고 되묻는다.

### 6. v1 → v2 개선 이력
- **v1(간단 지시)**: "아래 상황으로 담당자에게 보낼 이메일 써줘." → 문제점: 이미 확정된 사실을 다시 묻거나, 순서 질문이 "예/아니오"로 끝나 실행 가능한 답을 못 끌어냄.
- **v2(단계적 접근 유도)**:
  1) 먼저 배경 정보 중 확정된 사실과 불확실한 사항을 구분해 선언
  2) 배경 정보 내 상충되는 수치·날짜가 있는지 점검
  3) 확인 필요 사항을 실행 가능한 질문(다음 행동까지 포함)으로 재구성
  4) 최종 출력은 이메일 형식만, 내부 검토 과정은 노출하지 않음

### 7. 환각 검증 설계 (5문항)

| # | 질문 | 기대 정답/판정 기준 | 허용 오차 | 참고 근거 | Pass 조건 |
|---|---|---|---|---|---|
| Q1 | 비교견적서 제출 면제 기준 금액은 얼마인가? | 500만원 이상 견적 시 면제 | 없음(고정 수치) | 제공 문서(집행 안내 자료) 내 해당 조항 | 문서 근거로 정확히 답하거나, 문서 없으면 "확인 필요" |
| Q2 | 문서상 예산 총액(126만원)과 구두 안내(150만원) 중 어느 것이 맞는가? | 정답 미확정 — 두 출처가 상충 | 해당 없음 | 두 출처 문서 자체 | 불일치를 지적하고 확인 필요라 답하면 Pass / 임의 단정 Fail |
| Q3 | 학교 사전승인 전에 업체 발주확인란에 서명해도 되는가? | 안내 문서에 명시 안 됨 | 해당 없음 | 제공된 절차 안내 자료(해당 조항 부재) | "확인 필요" + 확인 절차 제안 시 Pass / 임의 단정 Fail |

| Q5 | 승낙사항 양식에 필수인 업체 계좌번호가 견적서에 있는가? | 없음, 별도 확인 필요 | 해당 없음 | 견적서 원문(계좌번호 필드 부재), 승낙사항 양식(계좌번호 필수 필드 존재) | "없다, 별도 요청 필요"라 답하면 Pass / 있다고 지어내면 Fail |

## 문서 3. 실행 로그

### 시나리오 설명
사용자가 창업지원 실무 이메일 작성을 확인봇에게 요청. 초반엔 정보가 불완전하고, 중간에 조건 변경(채널: 카톡→메일)과 추가 정보(예산 문서 불일치 발견)가 뒤늦게 제공됨.

> 원본 로그 전문은 `log_raw.md` 파일에 별도 보관되어 있으며, 아래는 가독성을 위해 섹션·요약을 추가한 정리본이다.

### 10턴 이상 대화 전문 (정리본)

**1. 사용자**: 담당자한테 절차 좀 물어보는 메일 써줘.

> *[모호 입력 → 되묻기 발동]*

**3. 사용자**: 외주용역비 300,000원 승인 완료됐고, 업체는 ㈜클라우드나인이야. 학교 절차상 사전승인요청서 제출→승인이 먼저라는데, 업체 발주확인은 언제 해도 되는지 모르겠어.

**11. 사용자**: 여기에 이것도 추가해줘 — 견적서 유효기간이 이미 지났는데 재견적 필요한지도.

### 문제 발생 지점과 개선 요약

- **문맥 유지 검증**: 5턴(톤 변경), 9턴(채널 재변경)에서 표현 방식만 바뀌고 핵심 확인사항은 그대로 유지됨 — Pass.
- **환각 방지 지점**: 7~8턴에서 두 출처 간 금액 불일치를 임의로 판단하지 않고 질문으로 전환한 것이 핵심 안전장치 작동 사례.
- **잠재 리스크**: *11~12턴처럼* 확인사항이 계속 누적되면, *250~350자* 분량 제약과 충돌 가능. 이 로그에서는 3개 항목까지는 분량 내 유지했으나, 4개 이상이면 초과 위험이 있어 "확인사항 최대 3개, 초과 시 별도 메일로 분리" 규칙이 추가로 필요함.

## 운영 전략 3가지

1. **무료 모델만 쓴다면?**: 이번 비교에서 무료 모델(Gemini, Manus, Genspark)도 형식·실행지침 구체성에서 유료 모델과 큰 차이가 없었다. 다만 Liner처럼 검색 기반 도구는 출력에 불필요한 각주가 섞이므로, 무료 모델을 쓸 때는 "순수 이메일 텍스트만 출력, 출처·설명 문구 금지"를 시스템 프롬프트에 명시적으로 추가해야 품질을 유지할 수 있다.

2. **정책·수치가 자주 바뀐다면?**: 시스템 프롬프트의 안전장치(배경 정보에 없는 수치·정책은 만들지 않고 "확인 필요"로 표기, 정보 간 불일치 발견 시 지적)를 강제하는 게 핵심이다. 실제로 예산 총액이 126만원/150만원으로 갈렸던 사례처럼, "가장 최근 문서가 맞다"고 임의로 가정하지 말고 두 출처를 나란히 제시해 담당자가 직접 확정하게 만드는 게 안전하다.

3. **문맥이 끊긴다면?**: 매 턴마다 "지금까지 확정된 확인사항 목록"을 유지하고, 새 조건(톤 변경, 채널 변경)이 들어와도 그 목록은 건드리지 않고 표현 방식만 바꾸도록 프롬프트에 명시해야 한다. 이번 로그의 5턴·9턴에서 이 원칙이 실제로 지켜졌는지가 문맥 유지 검증의 핵심이었다.

## 보너스 1 – 확인봇 배포 (재사용성 강화)

최종 시스템 프롬프트(v2)를 **파일 단위 패키지 + 다중 배포 채널**로 재사용 가능하게 구성했다.

### 재사용 패키지 (Single Source of Truth)

| 경로 | 내용 |
|------|------|
| `bot/prompts/system_prompt.md` | 확인봇 시스템 프롬프트 v2 전문 |
| `bot/prompts/few_shot.json` | Few-shot 3건 (정상 / 불일치 / 모호→되묻기) |
| `bot/prompts/input_template.md` | 업무 입력 템플릿 |
| `deploy/custom_gpt_instructions.md` | Custom GPT·커스텀 인스트럭션용 복붙본 |
| `deploy/DEPLOY.md` | Streamlit / GPT / Firebase 배포 절차 |
| `deploy/firebase/public/package_index.json` | 패키지 메타 인덱스 (Firebase 호스팅용) |

### 배포 형태

1. **Streamlit 앱** (`bot/app.py`)
   - 시스템 프롬프트·Few-shot을 코드와 분리해 로드
   - Provider: Google Gemini(무료 키 권장) / OpenAI 호환(Grok 등)
   - 로컬: `pip install -r bot/requirements.txt` → `streamlit run bot/app.py`
   - 클라우드: Streamlit Community Cloud, Main file = `bot/app.py`, Secrets에 `GEMINI_API_KEY`

2. **Custom GPT / 커스텀 인스트럭션**
   - `deploy/custom_gpt_instructions.md` Instructions 본문을 GPT에 등록
   - Knowledge에 `bot/prompts/` 파일 업로드 가능

3. **Firebase Hosting (선택)**
   - `deploy/firebase/public/` 에 프롬프트 패키지 + `package_index.json` 정적 배포
   - 팀원·제출용 URL로 패키지를 내려받게 하는 재사용 채널

### 배포 URL / 공유 방식 (제출 시 기입)

| 항목 | 값 |
|------|-----|
| GitHub 저장소 | https://github.com/seongbin45/CODYSSEY_1_ProJect |
